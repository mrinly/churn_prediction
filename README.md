# Customer churn prediction

## Project overview

Задача - построить модель бинарной классификации, которая предсказывает вероятность ухода клиента телеком-оператора.

С точки зрения бизнеса удержать существующего клиента дешевле, чем привлечь нового. Если модель заранее распознаёт клиентов с высоким риском ухода, компания может сфокусировать удерживающие кампании (скидки, звонки, персональные предложения) именно на них, а не тратить бюджет на всю базу.

**Метрика соревнования:** ROC-AUC - устойчива к дисбалансу классов и оценивает качество ранжирования клиентов по вероятности оттока.

## Dataset description

Датасет из Kaggle-соревнования в рамках обучения вDLS ФПМИ МФТИ. Это телеком-данные: каждый объект - это один клиент, целевая переменная `Churn` (1 — клиент ушёл, 0 — остался).

Признаки делятся на три группы:

| Группа | Примеры | Тип |
|---|---|---|
| Демография | `Sex`, `IsSeniorCitizen`, `HasPartner`, `HasChild` | Категориальные |
| Услуги | `HasPhoneService`, `HasInternetService`, `HasOnlineSecurityService`, `HasTechSupportAccess`, `HasMovieSubscription` | Категориальные |
| Контракт и платежи | `ClientPeriod`, `MonthlyCharges`, `TotalSpent`, `HasContractPhone`, `PaymentMethod`, `IsBillingPaperless` | Смешанные |

Целевая переменная несбалансирована: доля класса 1 ниже 50%, но дисбаланс можно назвать умеренным.

## Methodology

### 1. EDA и предобработка

- **Обзор данных:** проверила размер, типы колонок, пропуски и случайные строки, чтобы понять структуру датасета
- **Анализ числовых признаков:** построила распределения `ClientPeriod`, `MonthlyCharges`, `TotalSpent`; выявила выбросы
- **Анализ категориальных признаков:** посмотрела частоты значений и доли оттока внутри каждой категории, чтобы понять, какие признаки визуально разделяют классы
- **Распределение целевой переменной:** подтвердила дисбаланс классов - это повлияло на выбор метрики (ROC-AUC вместо accuracy) и подход к обучению
- **Чистка данных:**
  - `TotalSpent` пришёл как строка с пропусками (у новых клиентов поле пустое), сконвертировала в число, пропуски заполнила медианой
  - Проверила датасет на дубликаты и некорректные значения

### 2. Baseline - Logistic Regression через Pipeline + GridSearchCV

Собрала воспроизводимый пайплайн, чтобы preprocessing и модель обучались на одних и тех же фолдах кросс-валидации без утечек данных:

```
Pipeline
  ├── ColumnTransformer
  │     ├── числовые признаки → StandardScaler
  │     └── категориальные признаки → OneHotEncoder(handle_unknown='ignore')
  └── LogisticRegression(class_weight='balanced', max_iter=1000)
```

Ключевые решения:
- **`StandardScaler` для числовых признаков** - логистическая регрессия чувствительна к масштабу, без нормализации коэффициенты неинтерпретируемы и сходимость ухудшается
- **`OneHotEncoder` с `handle_unknown='ignore'`** - если на тесте встретится категория, которой не было на трейне, модель не упадёт
- **`class_weight='balanced'`** - автоматически взвешивает классы обратно пропорционально частоте, чтобы модель не игнорировала меньший класс
- **`GridSearchCV` с 5 фолдами и `scoring='roc_auc'`** - перебрала `C ∈ {0.001, 0.01, 0.1, 1, 10, 100}`, выбрала параметр, максимизирующий ROC-AUC на кросс-валидации
- **`refit=True`** - после подбора лучшего `C` модель переобучилась на всём трейне (не только на фолдах), чтобы итоговая модель использовала максимум доступных данных

**Результат baseline:** ROC-AUC ≈ 0.84 (среднее по 5 фолдам).

### 3. Градиентный бустинг - CatBoost

Выбор модели обоснован составом данных: много категориальных признаков с разным числом значений.

- **Почему CatBoost:** в отличие от sklearn-моделей, CatBoost работает с категориальными признаками напрямую через параметр `cat_features` - не нужен OneHotEncoder
- **CatBoost default:** сначала зафиксировала качество модели с заданными по дефолту параметрами (ROC-AUC ≈ 0.83), затем начала подбор гиперпараметров
- **Ручной двойной цикл по `n_estimators` и `learning_rate`:** перебрала сетку 3×3 для быстрого понимания, в какой области параметров модель ведёт себя лучше - прежде чем запускать более дорогой Optuna

### 4. Optuna - байесовская оптимизация гиперпараметров

После ручного перебора перешла к оптимизации через Optuna. Пространство поиска:

| Параметр | Диапазон | 
|---|---|
| `iterations` | 100–1000 | 
| `learning_rate` | 0.01–0.3 | 
| `depth` | 3–8 | 
| `l2_leaf_reg` | 0.1–10.0 |

- **Стратифицированное разбиение (`stratify=y`)** - сохраняет долю классов в трейне и валидации одинаковой, иначе при дисбалансе оценка на маленьком валидационном куске была бы шумной
- **`auto_class_weights='Balanced'`** - тот же смысл, что у `class_weight='balanced'` в sklearn: не даём модели игнорировать меньший класс

### 5. Финальная модель

После Optuna взяла лучшие найденные гиперпараметры и переобучила модель на всём доступном датасете (без разбиения на train/val). Логика - на этапе подбора гиперпараметров разбиение нужно, чтобы честно оценить качество; но когда параметры уже зафиксированы, финальная модель должна видеть максимум данных, чтобы минимизировать дисперсию предсказаний.

Предсказания на тестовом датасете сделаны через `predict_proba(...)[:, 1]` , использовала именно вероятности, а не бинарные метки, потому что в соревновании метрика ROC-AUC оценивает качество **ранжирования** клиентов, а не решения «ушёл / не ушёл» при фиксированном пороге.

## Results

| Модель | ROC-AUC | Метод подбора |
|---|---|---|
| Logistic Regression (baseline) | 0.845 | GridSearchCV по `C`, 5-fold CV |
| CatBoost (default) | 0.826 | Без тюнинга |
| CatBoost (ручной tuning) | 0.830 | Ручной перебор 3×3 |
| CatBoost + Optuna | ~0.85| Байесовская оптимизация, 4 параметра |

Финальный `my_submission.csv` собран по предсказаниям CatBoost с лучшими параметрами из Optuna.

## Business Insights

- Ключевые факторы риска оттока по важности признаков модели: тип контракта (у помесячного контракта заметно выше риск ухода, чем годовой/двухлетний), срок обслуживания (`ClientPeriod`), величина ежемесячного платежа
- Клиенты без дополнительных услуг (`HasTechSupportAccess`, `HasOnlineSecurityService`) уходят чаще, возможная рекомендация бизнесу: предлагать такие услуги в пакете как инструмент удержания
